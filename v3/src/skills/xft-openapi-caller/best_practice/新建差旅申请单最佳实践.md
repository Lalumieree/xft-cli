# 新建差旅申请单最佳实践

## 适用场景

当用户希望新建差旅申请单，只提供了出发地、目的地、日期等业务信息，但没有提供表单名称或表单模板编号 `eventNumber` 时，优先使用本流程。

## 核心原则

- `eventNumber` 是表单模板固定编号，不是临时业务流水号。
- 不要猜测 `eventNumber`。
- 如果用户没有提供表单名称，不能直接按名称查询模板编号。
- 优先先查启用中的差旅申请模板列表，再判断是否可以唯一确定模板。

## 推荐流程

1. 先定位创建接口文档。

   - 查询词建议：`创建 差旅 申请单`
   - 目标接口：`新建差旅申请单（草稿、审批通过、审批中）`
2. 调用 `表单配置列表查询`，先只按类型查询启用中的差旅申请模板。

   - 接口地址：`https://api.cmbchina.com/itrip/xft-api/v1/form/query-event-list`
   - 示例请求体：

```json
{
  "start": 0,
  "limit": 100,
  "eventStatus": "1",
  "eventTypeList": ["TRAVEL_APPLY"]
}
```

3. 根据模板查询结果分支处理。

   - 如果只返回 1 个模板，直接使用该模板的 `eventNumber`。
   - 如果返回多个模板，不要猜测：
     - **列出所有可用的表单**，让用户选择一个；或
     - 让用户提供一张同模板旧单，再通过详情接口反查 `eventNumber`。
   - 如果返回 0 个模板，向用户说明当前企业下未查到启用中的差旅申请模板。
4. 在已知表单名称时，可走快捷分支。

   - 例如用户明确给出表单名称后，再按 `eventName + eventTypeList` 查询。
   - 示例请求体：

```json
{
  "start": 0,
  "limit": 20,
  "eventStatus": "1",
  "eventName": "福州auto申请",
  "eventTypeList": ["TRAVEL_APPLY"]
}
```

- 示例返回关键字段：

```json
{
  "body": {
    "total": 1,
    "list": [
      {
        "eventSeqId": 2153238995400170,
        "eventNumber": "01260319095833000001",
        "eventName": "福州auto申请",
        "eventStatus": "1",
        "eventType": "TRAVEL_APPLY"
      }
    ]
  },
  "returnCode": "SUC0000",
  "success": true,
  "message": "成功"
}
```

5. 模板无法直接唯一确定时，可用旧单详情作为回退方案。

   - 让用户提供一张同模板旧单的 `billId`。
   - 再调用差旅申请单详情接口，从返回结果中提取：
     - `eventNumber`
     - `eventName`
     - `aplEmpNumber`
   - 仅在模板已确认的前提下复用这些字段。
6. 模板确定后，再解析城市编码。

   - 示例命令：
   - `python scripts/resolve_xft_city_code.py --from-name "福州市鼓楼区" --to-name "上海市" --json`
   - 已验证示例：
     - `福州市鼓楼区 -> 4824`
     - `上海市 -> 73`
7. 用确定下来的 `eventNumber` 组装建单请求体(用户未指定时，单据状态优先 `APPRV_PEND`)。

   - 示例请求体：

```json
{
  "eventNumber": "01260319095833000001",
  "empNumber": "0120260012",
  "billStatus": "APPRV_PEND",
  "customFieldValues": [
    {
      "fieldNumber": "C8844024",
      "fieldValue": "出差"
    }
  ],
  "businessTrip": {
    "businessTripDetails": [
      {
        "departCityCode": "4824",
        "destinationCityCode": "73",
        "beginTime": "2026-03-25 00:00",
        "endTime": "2026-03-26 00:00",
        "roundTrip": "Y"
      }
    ]
  }
}
```

8. 先 `--dry-run`，确认后再真实调用创建接口。

   - 创建接口地址：`https://api.cmbchina.com/itrip/xft-api/v1/bills/apply/travel/create`
   - 默认优先创建 `APPRV_PEND` 草稿单，除非用户明确要求直接发起。
9. 创建完成后，立即调用详情接口校验。

   - 至少确认：
     - 已返回新单据编号
     - `billSts = 0`（草稿）
     - 表单名称、行程、日期与预期一致

## 已验证样例

### 样例一：已知表单名称

- 已验证接口：`表单配置列表查询`
- 已验证表单名称：`福州auto申请`
- 已验证表单编号：`01260319095833000001`

### 样例二：创建并校验成功

- 已验证新建单据编号：`2026032034885110`
- 已验证详情状态：`billSts = 0`
- 已验证行程：`福州市鼓楼区 -> 上海市`
- 已验证日期：`2026-03-25` 到 `2026-03-26`

## 操作规则

- 未知表单名称时，先查 `TRAVEL_APPLY` 启用模板列表，不要直接按名称查询。
- 模板有多个候选时，优先补问表单名称，而不是盲目选第一个。
- 如果用户无法提供表单名称，可让用户提供一张同模板旧单作为回退输入。
- 如果表单还有额外必填自定义字段，仅补问缺失字段，不要让用户重述全部需求。
- 用户未指定单据状态时，优先创建状态为"审批中"的单据
- 对创建、提交、变更、取消等有副作用的接口，在用户尚未确认关键载荷时，先 `--dry-run`。
